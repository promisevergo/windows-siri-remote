# Apple TV A2854 Windows 遥控器映射软件需求规格说明书

| 项目 | 内容 |
| --- | --- |
| 文档版本 | V1.6 |
| 文档日期 | 2026-07-22 |
| 目标设备 | Apple TV Siri Remote A2854（USB-C 版） |
| 目标平台 | Windows 10/11 64 位 |
| 软件形态 | 无图形界面的 Windows 控制台程序 |

## 1. 项目概述

### 1.1 项目目标

开发一款无图形界面的 Windows 控制台程序，将 Apple TV A2854 遥控器映射为 Windows 输入控制器，实现：

- 使用 Clickpad 触摸区域控制鼠标移动；
- 使用 Clickpad 中心按键执行鼠标左键单击、双击，以及长按发送 Enter 键；
- 长按 TV/控制中心键并配合 Clickpad 滑动，实现鼠标左键按住拖拽；
- 使用 Clickpad 圆环旋转控制鼠标滚轮；
- 使用双指轻碰 Clickpad 圆环区域执行鼠标右键单击；
- 使用圆环四方向按键发送 Windows 上、下、左、右方向键；
- 第二阶段支持按住语音键接收遥控器音频，并将音频转发到用户已安装的虚拟声卡；
- 使用返回键、TV/控制中心键、静音键和音量键执行指定的 Windows 操作。

### 1.2 产品范围

项目分两个阶段实施：

- **第一阶段：按键与触摸映射**。实现鼠标移动、鼠标按键、滚轮、方向键、Enter、Delete、Alt+Tab、静音和音量映射。
- **第二阶段：语音音频转发**。接收并解码 A2854 的麦克风音频，将音频实时写入用户已安装的虚拟声卡，例如 VB-CABLE。目标应用通过虚拟声卡的录音端点接收该音频。

软件以普通用户权限在 Windows 桌面环境运行，通过控制台显示状态和接收必要配置，不提供 WPF、WinUI、托盘或其他图形界面。软件不开发自定义蓝牙驱动、虚拟 HID 驱动或虚拟音频驱动。第一阶段应能够独立交付和使用，第二阶段不得影响第一阶段功能。

以下内容不属于本期范围：

- 自行开发、安装或捆绑虚拟声卡驱动；
- 对遥控器音频执行语音识别、转写或文字输入；
- 将遥控器直接注册为新的 Windows 系统麦克风设备；
- 使用 Clickpad 中心按键本身按住并拖拽对象；中心按键长按已固定映射为 `Enter`，鼠标拖拽统一通过长按 TV/控制中心键完成；
- 控制 Windows 登录界面、锁屏界面或 UAC 安全桌面；
- 未在本文件中定义的电源键、播放/暂停键映射；
- 多套按键方案、按应用切换方案或宏命令。

## 2. 适用平台与运行条件

### 2.1 适用平台

- Windows 11 64 位，作为主要开发和验收平台；
- Windows 10 22H2 64 位，作为兼容性验证平台。Windows 10 已结束常规支持，本项目仅承诺软件兼容性，不承诺操作系统安全更新或厂商支持；
- 支持 Bluetooth Low Energy（BLE）的电脑；
- Apple TV Siri Remote A2854。

### 2.2 前置条件

- A2854 已与 Windows 完成蓝牙配对并处于可连接状态；
- 首次使用时，控制台列出已配对且可识别的 A2854，由用户输入序号完成选择；软件记住设备标识，后续启动自动尝试连接，不要求重复选择；
- 第一阶段要求软件能够读取遥控器的按键和触摸数据；
- 第二阶段要求软件能够读取并解码遥控器的语音音频数据；
- 使用第二阶段功能前，用户已安装并启用兼容的虚拟声卡软件，例如 VB-CABLE；
- 目标应用已选择虚拟声卡的录音端点作为麦克风输入；
- 需要接收模拟鼠标或键盘输入的目标应用与本软件处于相同或更低的 Windows 权限级别。

### 2.3 术语

- **Clickpad**：A2854 顶部圆形触摸点击区域，由中心区域和外侧圆环组成。
- **中心区域**：Clickpad 中间可按压、可触摸的区域。
- **圆环区域**：Clickpad 外侧支持四方向点击和旋转手势的环形区域。
- **一次触摸会话**：从手指首次接触 Clickpad 到所有手指离开 Clickpad 的完整过程。
- **虚拟声卡播放端点**：本软件写入音频的 Windows 播放设备；以 VB-CABLE 为例，对应 `CABLE Input`。
- **虚拟声卡录音端点**：目标应用选择的麦克风设备；以 VB-CABLE 为例，对应 `CABLE Output`。
- **音频转发会话**：从用户按下语音键开始，到松开语音键、遥控器断开或会话被取消为止的音频接收、解码和转发过程。

## 3. 总体功能要求

### 3.1 启动与运行

1. 软件启动后应连接并监听用户上次选择的 A2854；首次使用或原设备不可用时，应允许用户重新选择设备。
2. 遥控器可用时，软件应启用当前已交付阶段的功能。
3. 遥控器断开时，软件不得继续产生鼠标、键盘或音量事件。
4. 遥控器重新连接后，软件应恢复事件监听，无需用户重启软件。
5. 软件关闭后应停止全部映射和音频转发。
6. 用户按下 `Ctrl+C`、关闭控制台窗口或系统通知进程结束时，软件应先释放模拟按键、鼠标、音频和设备连接资源，再退出。

### 3.2 控制台交互

1. 直接运行 `RemoteMapper.exe` 时，软件读取已有配置并启动映射；没有有效设备配置时，进入设备选择流程。
2. 设备选择流程在控制台列出候选设备的序号、名称和可区分标识，用户输入序号后保存选择并开始连接。
3. 如果只检测到一个符合条件的 A2854，可自动选择并保存；检测到多个设备时必须由用户选择。
4. 控制台默认只输出启动、连接、断开、重连、配置和错误状态，不持续输出每个按键或触摸采样，避免影响运行。
5. 支持 `--list-devices` 列出候选遥控器后退出，支持 `--verbose` 输出调试事件。
6. 第二阶段启用后，支持 `--list-audio-devices` 列出可用播放端点；用户选择的端点标识保存在配置文件中。
7. 启动失败或发生不可恢复错误时，应向标准错误输出明确原因、将错误写入日志，并以非零退出码结束。

### 3.3 事件处理原则

1. 每个物理按键的按下和松开状态应被正确区分。
2. 同一物理事件不得被重复映射为多个意外操作。
3. 遥控器断开、软件退出或输入流异常结束时，应清除全部未释放的按键和触摸状态。
4. 触摸移动使用相对位移，不将 Clickpad 坐标直接映射为屏幕绝对坐标。

### 4.1 Clickpad 中心按键

#### 4.1.1 单击

- 中心按键按下后在 1 秒内松开时，应生成一次鼠标左键单击。
- 为区分长按，中心按键按下时不得立即发送鼠标事件；在 1 秒内松开后，应立即发送左键按下和左键松开事件，不再额外等待双击判断窗口。

#### 4.1.2 双击

- 用户在 Windows 当前双击时间阈值内连续点击中心按键两次时，应被 Windows 识别为鼠标左键双击。
- 双击速度遵循 Windows 系统的鼠标双击速度设置，软件不单独维护另一套双击时间。

#### 4.1.3 长按

- 用户按住中心按键达到 1 秒时，应发送一次键盘 `Enter` 键按下和松开事件。
- 每次长按最多发送一次 `Enter` 键输入，继续按住不得重复发送。
- 长按已触发 `Enter` 后，松开中心按键时不得产生鼠标左键单击事件。

### 4.2 圆环四方向按键

- 按下圆环上、下、左、右方向时，分别发送对应 Windows 方向键按下事件。
- 松开物理方向键时，发送对应 Windows 方向键松开事件。
- 长按时应按 Windows 当前键盘重复延迟和重复速度持续发送方向输入，松开后立即停止。

### 4.3 其他实体按键

- 返回键固定映射为 `Delete`，不是 `Backspace`。
- TV/控制中心键按下时先开始计时，不立即产生 Windows 输入。
- TV/控制中心键在 1 秒内松开时，按 `Alt` 按下、`Tab` 按下/松开、`Alt` 松开的顺序执行一次 `Alt+Tab`，切换到下一个软件。
- TV/控制中心键持续按住达到 1 秒时，发送一次鼠标左键按下事件并进入拖拽状态，不再执行 `Alt+Tab`。
- 拖拽状态期间，Clickpad 上任意起点的单指滑动均控制鼠标移动，不触发圆环滚轮或双指右键手势。
- 松开 TV/控制中心键时，应立即发送鼠标左键松开事件并结束拖拽状态；每次长按最多生成一组鼠标左键按下和松开事件。
- 拖拽结束时应清除当前触摸坐标；如果手指仍停留在 Clickpad 上，必须等待手指全部离开后再开始新的普通触摸会话，避免鼠标跳动或误触。
- 静音键每次单击切换一次系统静音状态。
- 音量加、减键支持单击调节；长按期间由软件重复发送对应音量事件，松开后立即停止。

## 5. 触摸与旋转判定逻辑

### 5.1 判定目标

中心区域和圆环区域共同构成一个连续触摸面。软件必须根据一次触摸会话的首次接触位置和触点数量判定本次手势模式，避免鼠标移动、滚轮和鼠标右键同时触发。

TV/控制中心键长按进入拖拽状态时，拖拽规则的优先级高于本章普通触摸手势：Clickpad 单指触摸统一按鼠标移动处理，暂停圆环滚轮和双指右键识别，直至 TV/控制中心键松开并完成状态清理。

### 5.2 模式判定

一次触摸会话只允许完成以下一种映射：

| 首次接触位置与触点 | 模式或候选手势 | 后续行为 |
| --- | --- | --- |
| 一根手指从中心区域开始 | 鼠标移动模式 | 即使手指随后移动到圆环区域，仍继续控制鼠标移动 |
| 一根手指从圆环区域开始 | 圆环手势等待状态 | 单指产生有效旋转后进入旋转滚轮模式；第二根手指及时落入圆环后进入双指轻碰候选状态 |
| 两根手指均在圆环区域 | 双指轻碰候选状态 | 满足双指轻碰条件时执行鼠标右键单击 |
| 任一手指从中心区域开始 | 不得识别为双指轻碰圆环 | 按鼠标移动模式处理，或在无法保持单指轨迹时取消本次手势 |

### 5.3 状态流程

```text
无触摸
  ↓ Touch Down
读取首次接触位置
  ├─ 中心区域 → 锁定鼠标移动模式
  └─ 圆环区域 → 进入圆环手势等待状态
                    ├─ 单指产生有效旋转 → 锁定旋转滚轮模式
                    └─ 第二根手指落入圆环 → 双指轻碰候选状态
  ↓
持续处理当前模式
  ↓ Touch Up
判断候选手势或结束当前模式，然后清除状态
```

### 5.4 鼠标移动模式

1. 使用相邻触摸采样点的 X/Y 坐标差计算鼠标相对位移。
2. 手指在中心和圆环之间移动时不得切换为滚轮模式。
3. 手指离开后停止鼠标移动并清除上一个坐标。
4. 轻微触摸抖动不得造成明显的鼠标漂移。
5. 鼠标移动方向应与手指移动方向一致。

### 5.5 旋转滚轮模式

1. 仅使用单指相邻触摸采样点相对 Clickpad 中心的角度差判断旋转方向和距离。
2. 顺时针旋转映射为鼠标滚轮向下；逆时针旋转映射为鼠标滚轮向上。
3. 滚轮步数应随累计旋转角度增加，不因单个微小抖动频繁触发。
4. 手指在本次会话中滑入中心区域时，仍保持旋转滚轮模式。
5. 手指离开后结束滚轮模式，下一次触摸重新判断起点。

### 5.6 双指轻碰圆环模式

#### 5.6.1 有效手势条件

一次有效的双指轻碰圆环必须同时满足以下条件：

以下时间和距离阈值是第一版验收基线。第一阶段可行性验证完成后，可根据真实采样数据统一调整一次；进入核心功能开发前必须冻结并更新到本文档，避免开发和验收使用不同阈值。

1. 两个触点首次出现的位置都在预先标定的圆环区域内；
2. 任一触点都不得从中心按键区域开始；
3. 第二根手指应在第一根手指接触后的 100 ms 内出现；
4. 从第一根手指接触到两根手指全部离开的总时间不得超过 250 ms；
5. 每个触点的移动距离不得超过 Clickpad 有效宽度的 5%；
6. 两根手指的离开时间差不得超过 100 ms；
7. 整个过程中不得出现 Clickpad 中心按键 `Select` 的物理按下事件；
8. 两个触点在离开前始终位于圆环区域内，不得进入中心按键区域。

#### 5.6.2 输出规则

- 满足全部条件时，软件发送一次鼠标右键按下和一次鼠标右键松开；
- 每次触摸会话最多产生一次鼠标右键单击；
- 双指轻碰候选状态期间不得产生鼠标移动或滚轮事件；
- 任一条件不满足时，取消本次双指轻碰，不发送鼠标右键事件；
- 候选手势取消后，应等待两根手指全部离开，再开始识别下一次触摸会话。

## 6. 第二阶段：语音音频转发

### 6.1 实现目标与边界

第二阶段仅负责接收、解码和转发 A2854 的麦克风音频，不执行语音识别或文字输入。软件复用用户已安装的虚拟声卡，不自行创建虚拟麦克风或安装音频驱动。

以 VB-CABLE 为例：

- 本软件将解码后的音频写入播放端点 `CABLE Input`；
- 需要使用遥控器语音的目标应用选择录音端点 `CABLE Output` 作为麦克风；
- 虚拟声卡负责把播放端点收到的音频传递到录音端点。

### 6.2 正常流程

```text
用户安装并启用虚拟声卡
  ↓
目标应用选择虚拟声卡录音端点作为麦克风
  ↓
用户按下语音键
  ↓
软件创建音频转发会话
  ↓
接收并解码遥控器音频
  ↓
持续写入虚拟声卡播放端点
  ↓
用户松开语音键
  ↓
停止写入并结束音频转发会话
```

### 6.3 语音键按下

检测到语音键由松开变为按下时，软件应：

1. 确认用户选择的虚拟声卡播放端点可用；
2. 创建新的音频转发会话；
3. 初始化遥控器音频接收和解码；
4. 打开虚拟声卡播放端点；
5. 开始将解码后的音频持续写入虚拟声卡；
6. 将内部状态切换为“正在转发音频”。

按键保持按下期间重复收到的按下事件不得创建多个音频转发会话。

### 6.4 音频处理与转发规则

- 软件应将遥控器的压缩音频解码为虚拟声卡支持的 PCM 格式；
- 如果解码格式与虚拟声卡要求不一致，应进行必要的采样率、位深或声道转换；
- 音频应在用户按住语音键期间连续写入虚拟声卡，不等待整段录音完成；
- 软件应允许用户从已安装的 Windows 播放设备中选择虚拟声卡播放端点，不硬编码设备名称；
- 默认不把音频保存为本地文件；
- 音频不得上传到网络服务，也不得传递给语音识别模块；
- 音频转发不得产生键盘文字或自动发送 Enter。

### 6.5 语音键松开

检测到语音键由按下变为松开时，软件应：

1. 停止接收本次遥控器音频；
2. 将已解码的少量剩余音频写入虚拟声卡；
3. 停止并关闭虚拟声卡播放流；
4. 结束并释放本次音频转发会话；
5. 将内部状态恢复为空闲。

### 6.6 虚拟声卡依赖

- 虚拟声卡由用户自行安装、配置和维护；
- 软件至少应以 VB-CABLE 作为第二阶段验收环境；
- 未安装或未选择虚拟声卡时，软件应禁用音频转发并提示用户完成配置；
- 虚拟声卡不可用或转发失败不得影响第一阶段的按键和触摸映射功能。

## 7. 异常与边界情况

| 场景 | 预期处理 |
| --- | --- |
| 软件启动时遥控器未连接 | 不产生映射事件；等待遥控器重新连接 |
| 使用过程中遥控器断开 | 立即停止当前触摸、按键和音频转发会话，并释放所有状态 |
| 遥控器重新连接 | 自动恢复输入监听 |
| 语音键快速点击但没有有效音频 | 结束音频转发会话，不产生文字或按键输入 |
| 语音键按下时已有音频转发会话 | 忽略重复按下事件，不创建第二个会话 |
| 未收到语音键松开事件 | 遥控器断开或连续转发达到 60 秒时自动停止并结束本次会话 |
| 语音数据中断或无法解码 | 停止音频转发并提示遥控器音频处理失败；第一阶段映射继续运行 |
| 未安装、未选择或无法打开虚拟声卡 | 不启动音频转发，提示用户安装或选择可用的虚拟声卡播放端点 |
| 转发期间虚拟声卡被移除或失效 | 立即停止音频转发并释放播放流；第一阶段映射继续运行 |
| 触摸起点位于中心与圆环边界附近 | 按预设区域边界只判定一次，并锁定到手指离开 |
| 触摸过程中从中心跨到圆环 | 保持鼠标移动模式 |
| 单指已锁定旋转滚轮模式后从圆环跨到中心 | 保持旋转滚轮模式 |
| 双指轻碰时任一触点位于或进入中心区域 | 取消双指轻碰，不发送鼠标右键事件 |
| 双指轻碰期间任一触点移动超过阈值 | 取消双指轻碰，本次触摸会话不产生鼠标右键、滚轮或移动事件 |
| 双指轻碰期间物理按下 Clickpad | 按物理按键规则处理，不发送鼠标右键事件 |
| 第二根手指出现过晚或两根手指释放时间差过大 | 不识别为双指轻碰，不发送鼠标右键事件 |
| 触摸数据短暂丢失 | 不使用过期坐标计算大幅位移；恢复后以新采样点为起点 |
| 按键按下期间遥控器断开 | 补发相应松开状态，避免方向键或音量键持续按住 |
| TV/控制中心键拖拽期间遥控器断开或软件退出 | 立即补发鼠标左键松开事件，结束拖拽并清除触摸状态，避免鼠标左键持续按住 |
| TV/控制中心键达到长按阈值后继续收到重复按下事件 | 保持当前拖拽状态，不重复发送鼠标左键按下事件 |
| 软件退出 | 停止监听并释放全部按键、触摸、音频转发及连接资源 |

## 8. 非功能需求

### 8.1 性能

- 普通按键从接收到生成 Windows 输入事件的延迟目标不超过 50 ms；中心按键短按以“松开事件”为计时起点，50 ms 内生成鼠标单击；中心按键长按达到 1 秒后，应在 50 ms 内生成 Enter 键输入；TV/控制中心键长按达到 1 秒后，应在 50 ms 内进入鼠标左键拖拽状态；
- 触摸移动应连续，无明显跳动或肉眼可感知的额外停顿；
- 有效双指轻碰结束后应在 100 ms 内生成鼠标右键单击；
- 第二阶段中，语音键按下后应在 200 ms 内启动音频接收会话；收到第一个有效音频包后，应在 200 ms 内向虚拟声卡写入首批 PCM 数据；
- 第二阶段的音频转发应连续，不得出现明显的周期性卡顿或音频中断。

### 8.2 稳定性

- 软件连续运行 8 小时不应崩溃或出现持续的卡键、漂移；
- 遥控器连续断开和重连 10 次后，映射功能仍应正常；
- 单次按键、触摸、音频解码或虚拟声卡写入失败不得导致软件退出；
- 第二阶段发生故障时，第一阶段映射功能应继续正常运行。

### 8.3 易用性

- 软件启动后，在遥控器已配对的情况下无需重复设置即可使用；
- 连接、音频转发中和失败状态应有简洁、可区分的控制台文字提示；
- 第二阶段应通过控制台提供虚拟声卡播放端点选择，并记住用户上一次成功使用的设备；
- 错误提示应输出到控制台和日志，并说明失败原因及用户可执行的下一步操作。

### 8.4 兼容性

- 支持 Windows 10/11 64 位桌面环境；
- 第二阶段支持向 Windows 中已安装的兼容虚拟声卡播放端点写入音频；
- 第二阶段至少完成对 VB-CABLE 的兼容性验证；
- 不承诺支持管理员权限窗口、安全桌面、远程桌面特殊输入通道或使用专有输入机制的应用。

### 8.5 隐私与数据

- 默认不长期保存原始语音音频；
- 默认不在日志中记录原始音频；
- 音频仅在本机的遥控器连接与用户选择的虚拟声卡之间转发，不上传到网络服务；
- 软件不生成、保存或记录语音识别文字。

## 10. 开发优先级

### 10.1 第一阶段：可行性验证

在完整开发前先完成最小验证程序，确认目标 Windows 环境能够读取：

1. 全部实体按键的按下和松开事件；
2. Clickpad 单指和双指触摸坐标、触点数量及首次接触位置；
3. 圆环旋转所需的连续触摸数据。

以上三项是第一阶段的继续/暂停门槛。尤其是双指触点数量、原始坐标或圆环连续数据任一项无法通过普通 Windows API 稳定取得时，应先调整对应手势需求，不得直接进入完整控制台程序开发。

### 10.2 第一阶段：核心功能

1. 遥控器连接、断开与自动恢复；
2. 中心按键左键单击、双击和长按 1 秒映射 Enter 键；
3. Clickpad 鼠标移动；
4. 圆环旋转滚轮；
5. 双指轻碰圆环映射鼠标右键单击；
6. 圆环四方向键；
7. 返回、TV/控制中心短按 `Alt+Tab`、TV/控制中心长按拖拽、静音和音量按键映射；
8. 断线、卡键及输入流异常处理。

### 10.3 第一阶段：体验完善

1. 鼠标移动速度、抖动过滤和滚轮步进调优；
2. 控制台连接、重连和失败状态提示；
3. 连续运行、睡眠唤醒及重复重连稳定性优化；
4. 配置文件、命令行参数和自包含发布。

### 10.4 第二阶段：可行性验证

第一阶段完成后，使用最小验证程序依次确认：

1. 能够检测语音键按下和松开，或通过音频流可靠判断会话开始和结束；
2. 能够通过普通 Windows API 接收 A2854 的麦克风音频数据；
3. 能够确认音频分包、序号、采样参数和编码格式，并把遥控器音频解码为连续、可播放的 PCM；
4. 能够打开用户选择的虚拟声卡播放端点并写入测试音频；
5. 目标应用从虚拟声卡录音端点能够收到清晰、连续的遥控器语音。

如果无法通过普通 Windows API 读取或解码 A2854 音频，应暂停并重新评估第二阶段技术路线。不得以电脑自带麦克风替代遥控器麦克风并宣称第二阶段完成，且该结果不得阻塞第一阶段交付。

### 10.5 第二阶段：核心功能

1. 语音键控制音频转发会话开始和结束；
2. 遥控器音频接收、解码及必要的格式转换；
3. 向虚拟声卡播放端点实时、连续写入音频；
4. 虚拟声卡播放端点选择和配置保存；
5. 遥控器断开、虚拟声卡失效和音频解码失败处理；
6. 音频会话结束、软件退出及异常情况下的资源释放。

## 11. 建议开发环境与技术方案

### 11.1 开发环境

| 项目 | 建议 |
| --- | --- |
| 开发系统 | Windows 11 64 位；另准备一台 Windows 10 22H2 64 位设备做兼容性回归 |
| 开发工具 | .NET 10 SDK、Windows SDK；可使用 Visual Studio 2026 或 `dotnet` 命令行 |
| 运行时 | .NET 10 LTS |
| 目标框架 | `net10.0-windows10.0.19041.0` |
| CPU 架构 | 首版仅发布 x64 |
| 硬件 | A2854 遥控器、稳定的 BLE 适配器；第二阶段增加 VB-CABLE |
| 源码管理 | Git；提交构建所需的解决方案、锁定文件和构建说明 |
| 测试 | xUnit，用于按键状态机、长按判定、触摸手势、命令行参数和音频会话逻辑的自动化测试 |

发布版本采用自包含的 x64 单文件部署，用户无需额外安装 .NET 运行时。程序以普通未打包 Windows 进程运行，不要求 MSIX；可将可执行文件、默认配置示例和使用说明放在一个压缩包中发布。

### 11.2 推荐框架与 Windows API

| 功能 | 建议方案 |
| --- | --- |
| 程序宿主 | .NET 控制台程序，不引用 WPF、WinUI、Windows Forms 或嵌入式浏览器 |
| 控制台交互 | 使用 `System.Console` 和少量手写参数解析，不引入大型命令行框架 |
| 设备发现与授权 | `Windows.Devices.Enumeration` |
| BLE/GATT 通信 | `Windows.Devices.Bluetooth` 与 `Windows.Devices.Bluetooth.GenericAttributeProfile`；可行性验证必须确认未打包控制台进程能够完成所需设备访问 |
| 鼠标与键盘输出 | Win32 `SendInput`；方向键重复逻辑读取 Windows 当前键盘重复设置 |
| 静音与音量输出 | Windows 媒体虚拟键，通过 `SendInput` 发送 |
| 音频输出 | WASAPI；建议使用 NAudio 简化播放端点枚举、格式转换和流式写入 |
| 配置保存 | `%LOCALAPPDATA%\RemoteMapper\config.json`，仅保存设备标识、灵敏度、冻结后的手势阈值和音频端点标识 |
| 日志 | `%LOCALAPPDATA%\RemoteMapper\logs` 下的文本滚动日志，只记录连接状态、事件类型和错误，不记录触摸原始轨迹或音频内容 |
| 退出处理 | `Console.CancelKeyPress` 配合 Win32 控制台结束信号处理，统一释放所有按键和连接状态 |

第二阶段在确认 A2854 的真实音频编码前，不预先绑定某个 Opus 解码库。只有验证音频确为标准 Opus 后再选择维护中的解码库；若为 Apple 私有变体，应先完成协议和授权风险评估。

### 11.3 建议代码结构

为保持简单可维护，建议只拆分为三个项目：

```text
RemoteMapper.Cli
  控制台入口、命令行交互、程序生命周期和 Windows API 适配
RemoteMapper.Core
  遥控器事件模型、按键/触摸状态机、映射规则和音频会话接口
RemoteMapper.Tests
  Core 的自动化测试
```

蓝牙读取层应通过一个简单接口向 `Core` 输出统一的按键、触摸和音频包事件。这样即使可行性验证后发现 A2854 需要从 GATT 改用其他 Windows 设备接口，也无需重写手势和映射逻辑。控制台入口不直接处理原始蓝牙包，也不直接调用 `SendInput`。

### 11.4 开发顺序建议

1. 先制作最小设备探测控制台程序，保存实体按键、触点数量、坐标和 GATT 特征的脱敏日志；
2. 通过第一阶段继续/暂停门槛后，实现纯状态机及自动化测试；
3. 接入 `SendInput`，完成第一阶段映射和断线释放；
4. 补充控制台设备选择、状态输出、配置保存、命令行参数和单文件发布；
